Description du jeu

Nous vous proposons de découvrir Akka au travers d'un jeu autour de la gestion de trafic d'un aéroport.

Vous êtes un contrôleur aérien. Des avions vont vous envoyer une série de message pour pouvoir attérir.

Vous avez une liste de piste d'attérissage à votre dispotion. A vous de bien les utiliser dans cet univers hautement concurrentiel pour arriver à se faire poser tous les avions.

Le serveur de jeu se chargera ensuite d'envoyer les avions jusqu'à la porte d'embarquement.

Le workflow est simple

Incoming

Un avion arrive et vous contacte. Vous devez lui répondre par new Land(...) avec la référence vers une piste d'attérissage.

HasLanded

Quand l'avion est au sol, il vous demande l'adresse de la tour de contrôle au sol. Répondez lui tout simplement Contact(groundControl). ATTENTION, l'avion est toujours sur la piste.

HasLeft

L'avion a liberé la piste d'attérissage.

La liste intégrale des états des avions est disponibles ici. Seul les trois états cités ici sont intéressants.

La liste des commandes est disponible ici ici. Seul les citées ici sont intéressantes.

Nous sommes toujours dans le code de la classe :AirTrafficControl.scala.

Un acteur est un objet, il peut avoir un état. Ici, l'agent de contrôle va devoir se souvenir des affectations qu'il fait. Pour cela créer un variable d'instance de type Map

var allocations = Map.empty[ActorRef,ActorRef]
dans la classe. La clé est la référence de l'avion, la valeur est la référence vers la piste d'attérissage affectée.

Lors de l'arrivée d'un avion, il suffit de connaitre la liste des pistes libres, en Scala, cela s'obtient par la ligne suivante

Si cette liste est non vide

Récupérez le premier élément, envoyer l'ordre à l'avion d'attérir sur cette piste, et stocker l'association dans la Map.

allocations += (plane -> freeRunway)

Si cette liste est vide

Nous ne sommes pas obliger de répondre tout de suite. Nous sommes en environnement totalement asynchrone. Nous pouvons mettre de côté cette demande et la traiter quand une place sera libérée. Pour cela, il faut créer variable d'instance de type Queue:

Cette queue contiendra la liste des avions en attentes avec la ligne de code suivante:

S'il y a de l'attente

NB

Un acteur est encapsulé, il n'est donc pas nécessaire d'utiliser des structures de données concurrentes. Un acteur ne sera toujours accéder que par 1 seul thread au plus.

Une bonne pratique est d'utiliser des références mutables sur des structures immutables.En effet, les structures immutables peuvent être échangées avec d'autres acteurs sans problème d'état partagé.

Etape 3: Radio is failing (30 min)

Cette fois, l'avion va 'perdre' des messages. Il vous faut vous assurer qu'il a bien compris vos ordres. Un des patterns classiques pour cela en message est celui des Ack.

Un acteur n'est pas coûteux en mémoire. On peut en créer qui ont une durée de vie très courte. De plus, il est préférable de garder des responsabilités ciblées pour chaque acteur. Planification du timeout

Créer un case object Timeout au sein du même fichier de classe.

L'OrderSender va ensuite s'envoyer un message pour le prévenir du timeout de l'avion. Pour cela, créer un case object Timeoutdans votre fichier OrderSender.scala. Juste après l'envoie du message, nous pouvons utiliser le scheduler proposé par Akka pour s'envoyer un message:

context.system.scheduler.scheduleOnce(ackMaxTimeout, self, Timeout)

Comportement

L'implémentation de l'acteur est ensuite facile à réaliser. Si l'on recoit le message fr.xebia.xke.akka.airport.message.command.Ack, on arrête l'acteur. Si l'on recoit Timeout il faut de nouveau envoyer le message et planifier le Timeout.

def receive: Receive = {
    case Ack =>

        log.debug(s"Ack of ${msg.getClass.getSimpleName} to ${target.path.name}")

        context stop self

    case Timeout =>

        log.debug(s"Timeout of ${msg.getClass.getSimpleName} to ${target.path.name}")
        target ! msg
        import context.dispatcher
        context.system.scheduler.scheduleOnce(ackMaxTimeout, self, Timeout)
}

Etape 4: EventSourcing (bonus)

Cette étape permet de stocker l'état de l'AirTrafficControl. La communication peut être coupée à tout moment, il faut donc être capable de redémarrer un AirTrafficControl avec un état du jeu persisté.

Lors de cette étape du jeu, un ChaosMonkey peut être envoyé à tout moment à l'AirTrafficControl. A la réception de ce message, l'AirTrafficControl redémarre. Il pert alors son état. Le superviseur de cet acteur est AirportManager.scala. C'est lui qui fixe cet stratégie.

Eventsourcing

Contrairement à une approche classique à base d'ORM et d'update dans une base de données, cette approche ce fait en mode 'append-only'. L'objet stocke dans une structure de données des événements métiers (un avion a été affecté à une piste, un avion est ajouté en attente...).

Le système peut alors décider de faire rejouer tous les événements d'un acteur quand il 'renaît' avant de l'autoriser à consommer des messages de sa mailbox.

Mais il faut faire une différence entre un message en entrée de l'acteur (Command), et l'événement fonctionnel qui en résulte (Event). Il existe deux stratégie : CommandSourcing (persistance des messages en entrée) et EventSourcing (persistence des changements d'états).

Quels sont pour vous les commandes de l'acteur AirTrafficControl? Quels sont les événements? Les commandes sont:

  • Incoming
  • HasLanded
  • HasLeft
Les événements peuvent être:
  • PlaneAffected(plane,runway)
  • PlaneDisaffected(plane)
  • PlaneEnqueued(plane)
  • PlaneDequeued
Dans le même fichier que AirTrafficControl.scala, créer donc quatre classes:
  • case class PlaneAffected(plane:ActorRef,runway:ActorRef)
  • case class PlaneDisaffected(plane:ActorRef)
  • case class PlaneEnqueued(plane:ActorRef)
  • case object PlaneDequeued
  • case class Initiated(groundControl: ActorRef, runways: Set[ActorRef], ackMaxTimeout:Int)
Cette dernière classe correspond au message d'intialisation du l'ATC par le jeu (transparent pour vous jusque là!). Il ne sert à rien non plus de gérer le message ChaosMonkey. Il permet seulement de lancer une exception pour redémarrer l'instance.

EventsourcedProcessor

Pour utiliser l'EventSourcing, le code doit être changé en profondeur. Il faut tout d'abord changer d'héritage. Dans AirTrafficControl.scala faites hériter la classe d'EventsourcedProcessor. Vous pouvez garder le mixin avec ActorLogging. Cette nouvelle classe nécessite l'implémentation de deux méthodes:

  • override def receiveCommand: Receive
    • cette méthode receive va recevoir les commandes comme la méthode receive d'un acteur classique.
  • override def receiveRecover: Receive
    • cette méthode receive va recevoir les événements qui ont été persistés au redémarrage d'un acteur.

Aucun message ne sera traité par receiveCommand tant que receiveRecover recoit des messages. C'est l'abstraction qui gère cela pour nous.

receiveCommand

On peut copier-coller l'intégralité du traitement fait jusqu'ici depuis la méthode recover dans la méthode receiveCommand. Il faut cependant la modifier un peu:

Le code reste la même pour savoir si une piste est libre ou pas. Mais plutôt que d'envoyer un message à l'avion ou stocker l'affectation, il faut d'abord créer l'événement, le persister avant de faire des "effets de bord". Cela se fait grâce à la méthode

Une fois persisté, l'acteur appelle la fonction handler. Pour notre cas, cela donne:

        ensure(p,Land(f))
        allocations += (p -> f)

Nous voyons bien ici la différence qu'il existe entre la valorisation métier d'une commande et les effets de bords.

receiveRecover

Cette méthode doit être plus simple. Elle est appelée par le framework lors du démarrage d'un acteur. Ce qu'il est important de savoir, c'est que l'identifiant unique d'un acteur est son path dans le système. Au démarrage, le système vérifie si des événements on déjà été stockés pour cet identifiant. Si oui, il les joue un par un avant de laisser l'acteur consommer des messages de sa mailbox. Nous utilisons LevelDB comme implémentation de store. C'est celle par défaut.

Voici le code de gestion du recover: 

override def receiveRecover: Receive = {
    case PlaneAffected(plane, runway) =>
        allocations += (plane -> runway)

    case PlaneDisaffected(plane) =>
        allocations -= plane

    case PlaneEnqueued(p) =>
        pendings = pendings enqueue plane

    case PlaneDequeued =>
        val (_, newPendings) = pendings.dequeue
        pendings = newPendings

    case Initiated(g,r,a) =>
        this.groundControl = g
        this.runways = r
        this.ackMaxTimeout = a
}

Bonus: Supervisor

Un acteur est forcément supervisé par celui qui l'a créé. Cela forme un graphe d'objet, en partant du système comme niveau le plus haut.

Si un acteur lance une exception, son superviseur reçoit un message avec l'exception. Il peut alors décider du sort de ses 'supervisés'.

Un exemple se trouve dans la classe AirportManager.scala.

override def supervisorStrategy: SupervisorStrategy = OneForOneStrategy(loggingEnabled = true) {
    case _: ChaosMonkeyException =>
        SupervisorStrategy.Restart
}

Grâce au pattern matching, la gestion d'erreur est très claire. En cas d'une exception ChaosMonkeyException, le superviseur demande à l'acteur de faire un Restart.

Directive

Si vous inspectez le code de SupervisorStrategy, vous trouverez la liste des actions possibles:

  • Resume
    • l'acteur continue comme si de rien était.
  • Stop
    • l'acteur se termine et n'existe plus.
  • Restart
    • l'acteur est redémarré. Il pert le message qui a produit l'exception et perd son état.
  • Escalate
    • le superviseur ne sait pas quoi faire et escale le problème.

SupervisorStrategy

Il existe une autre variation du traitement d'erreurs:

  • OneForOneStrategy
    • la commande choisie par le superviseur ne s'applique qu'à l'acteur en erreur.
  • AllForOneStrategy
    • la commande choisie s'applique à tous les acteurs 'enfants' du superviseur.

Options

En paramètre de la stratégie, on retrouve les paramètres suivants:

  • maxNrOfRetries
    • nombre de fois que l'exception doit être lancée pour un même acteur avant que le superviseur n'intervienne.
  • withinTimeRange
    • fenêtre glissante de temps pour le compteur du nombre d'exceptions.
  • loggingEnabled
    • loggue ou non l'exception
Ces options permettent déclencher des actions qui si une exception est appelée, par exemple, 5 fois en moins de 10 secondes.

Bonus: Finite State Machine

Le modèle acteur est adapté à la gestion de machine à états finis. Nous ne l'avons pas vu ce soir, mais cette fonctionnalitée est beaucoup utilisé dans notre serveur Play!. Pour cela, nous utilisons une commande très puissante: become. La méthode become sur le context d'un acteur permet de définir le code de la méthode qui va traiter le prochain message.

Exemple

Un exemple de l'utilisation de cette fonctionnalité se trouve dans le fichier Runway.scala. En effet, une piste d'attérissage à deux états possibles, free ou occupied(plane).

free

Lorsqu'elle est libre, la piste d'atérissage accepte l'atérissage d'un nouvel avion. Elle devient alors occupied(plane).

occupied

Lorsqu'elle est occupée, la piste d'atérissage émet une erreur lorsqu'elle reçoit un nouvel avion et termine. Sinon, elle redevient free lorsqu'un avion s'en va.

L'apport du become permet d'isoler les différents états et évite de stocker des variables mutables sur l'acteur. Finalement, l'acteur ne pourrait contenir aucun état. Tout peut être passer d'appels en appels en invoquant la méthode become à chaque réception de message. Cela peut rendre le code plus facilement testable unitairement.

Bonus: Akka Cluster

Pour faire communiquer les acteurs du serveur Play! avec ceux du joueur, nous utilisons le module akka-cluster. Baser sur une communication TCP via Netty, il permet de créer une connexion entre deux acteurs systèmes distants. Un acteur peut alors discuter avec un autre, qui peut potentiellement être sur une autre machine.

A part le code d'infrastucture, il n'y a pas d'impact dans le code métier. Pour démarrer en mode cluster, il faut ajouter la dépendance vers le module akka-cluster dans le fichier build.sbt.

Ensuite, dans le fichier application.conf il faut ajouter la propriété suivante : akka.actor.provider="akka.cluster.ClusterActorRefProvider".

Ensuite, il est possible de s'enregistrer à un EventBus des événements qui vont se passer sur le cluster. Cela est fait dans la classe AirportLocator.scala avec Cluster(context.system).subscribe(self, classOf[ClusterDomainEvent]). La méthode receive reçoit alors tous les événements du cluster.

case MemberUp(member) =>
    //connexion d'un nouveau membre

case UnreachableMember(member) =>
    //la communication réseau est rompu

L'objet Member contient l'adresse du système et les rôles qu'il déclare accepter. Cela permet de créer des noeuds taggés (back, web, log...)

On peut alors faire deux choses, créer des acteurs sur un autre système de que le sien, ou faire du lookup par path. Ici nous utilisons du lookup. L'instance d'AirportManager est localisée par son path. 

context.actorSelection(ActorPath.fromString(address.toString) / "user" / airportCode)
